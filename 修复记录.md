## 一、本轮已修复（按优先级）

### 1. 构建阻断：废弃仓库 + SDK 版本冲突 ✅
- `build.gradle`
  - `allprojects.repositories`：移除已关停的 `jcenter()` 与 `dl.bintray.com/*`，改为 `google()` + `mavenCentral()` + 阿里云镜像（`maven.aliyun.com/repository/public`、`/google`）+ `jitpack.io`。
  - `project.ext`：统一 SDK 到 `compileSdk=30 / targetSdk=30 / buildTools=30.0.3 / minSdk=21 / versionCode=38`。
  - `buildscript.repositories`：补充阿里云镜像（解析 andserver 插件等）。
- `versions.gradle`
  - `build_versions` 对齐为 30/30/21/30.0.3（原为 28/28/15/28.0.3，且实际未被引用，纯清理）。
  - `addRepos()`：移除 `jcenter()` → `mavenCentral()`。
- `app/build.gradle`
  - `compileSdkVersion` / `minSdkVersion` / `targetSdkVersion` 改为引用 `rootProject.ext`，消除与模块（`rootProject.ext` 原 29/26）的冲突。
  - 协程 `kotlinx-coroutines-android` 1.3.9 → **1.5.2**（避免与新增 lifecycle-ktx 传递依赖 1.4.1+ 版本错配；与项目 Kotlin 1.5.31 匹配）。

### 2. 安全高危：fastjson RCE 漏洞 ✅
- 移除依赖 `com.alibaba:fastjson:1.1.72.android`（已知反序列化 RCE，CVE-2022-25845 等）。
- 新增 `com.google.code.gson:gson:2.9.0`，迁移全部调用点：
  - `PlaybackVideoFragment.kt`：`JSON.toJSONString(media)` → `Gson().toJson(media)`；`JSON.parseObject(body, DlanUrlDTO)` → `Gson().fromJson(...)`。原"序列化字符串相等比较"改为**按 `url` 比较**，避免不同序列化器格式差异导致每轮轮询都重播。
  - `WebController.kt`：`JSON.parseObject(data)` → `Gson().fromJson(data, JsonObject)`；`sd.getString(k)` → `sd.get(k)?.getAsString`。
  - `LiveListHolder.kt`：`JSON.toJSONString(a).equals(JSON.toJSONString(b))` → `Gson().toJson(a) == Gson().toJson(b)`（同序列化器，比较自洽）。

### 3. Manifest 安全配置 ✅
- `android:allowBackup="true"` → **`false`**（防止 adb 备份扒取应用数据）。
- 移除 `READ_PHONE_STATE`（全工程未使用）。
- 保留 `GET_TASKS`：`WindowHelper.setTopApp()` 用 `getRunningTasks` 把应用切到前台，仍依赖该权限（API 21+ 仅返回自身 task，行为受限但可用）。

### 4. 协程作用域泄漏（生命周期未绑定）✅
- `MainActivity.kt`：`CoroutineScope(EmptyCoroutineContext)` → `lifecycleScope`（延时启动前台服务）。
- `PlaybackVideoFragment.kt`：保留 `scope` 字段名，但改为 `private val scope: CoroutineScope = lifecycleScope`（匿名对象内仍可访问，且由 Fragment 生命周期自动取消）。**关键修复**：原 `CoroutineScope(EmptyCoroutineContext)` 无 Job，`onDestroyView` 里的 `scope.cancel()` 实际无效，导致 `initDlan()` 中 `while(true)` 死循环（每秒发送 DLNA 进度）在 Fragment 销毁后**永不停止** → 真泄漏；现由 `lifecycleScope` 在销毁时自动取消。
- 补充 `import androidx.lifecycle.lifecycleScope`（两处）。

### 5. WebServer 单例生命周期 ✅
- `WebServer.java`：原 `start()` 在局部变量创建 `Server`，丢弃引用、无 `stop()`，重复启动会在同端口崩溃。
  - 改为持有 `private Server server` 字段；`start()` 加 `synchronized` + 运行中则跳过（防重复启动）。
  - 新增 `stop()`（`server.shutdown()` 并置空）。
- `MainActivity.onDestroy()`：补 `App.webServer?.stop()`，退出时正确关闭局域网管理服务器。

### 6. 进度轮询过频（耗电）✅
- `MediaPlayerAdapter.kt`：`getProgressUpdatingInterval()` 返回 **16ms（≈62 次/秒）** → **1000ms**，与 DLNA 进度上报节奏一致，显著降低 CPU/耗电。

### 7. 构建阻断：AGP 4.2.2 与 Gradle 8.5 不兼容 → 向上升级构建链 ✅
- **报错**：`Could not create task ':app:mergeDebugAssets'` / `Cannot use @TaskAction annotation on method IncrementalTask.taskAction$gradle_core() because interface org.gradle.api.tasks.incremental.IncrementalTaskInputs is not a valid parameter to an action method.`
- **根因**：AGP 4.2.2 的 `mergeDebugAssets` 继承自内部 `IncrementalTask`，使用 `org.gradle.api.tasks.incremental.IncrementalTaskInputs`；该接口在 **Gradle 8.0 被彻底移除**（7.0 弃用、8.0 删除，改用 `InputChanges`）。AGP 4.2.2 官方仅支持 Gradle 6.7.1，无法在 Gradle 8.5 下配置。
- **决策：升级而非回退**。用户的 Gradle 环境已是 8.5，正确做法是把构建链抬到与 Gradle 8.5 匹配的现代版本，而不是把 Gradle 降回 6.7.1。最终配对（经官方兼容矩阵确认）：
  - **Gradle 8.5**（维持，≥ AGP 8.3 所需的 8.4）+ **AGP 8.3.2** + **Kotlin 1.9.22** + **JDK 17** + **compileSdk 34**。
- **andserver 插件的死结与处理**：工程还应用了 `com.yanzhenjie.andserver:plugin:2.1.9`（2019 老插件），依赖 **AGP 旧 Transform API**（AGP 7.0 / Gradle 7.0 已移除），升 AGP 后该插件必挂。→ **移除该 Gradle 插件**（`build.gradle` 的 `classpath` 与 `app/build.gradle` 的 `apply plugin`），但**保留运行时依赖** `com.yanzhenjie.andserver:api:2.1.9` + `kapt ...:processor:2.1.9`。该插件只是开发期 `runServer` 任务，移除不影响 APK 构建与网页投屏运行。
- **AGP 8 必需的配套改动（已落地）**：
  - 全部模块补 `namespace`（app=`com.hd.tvpro`；VideoUi=`com.google.android.exoplayer2.ui`；ffmpeg=`com.google.android.exoplayer2.ext.ffmpeg`；VideoPlayModule=`chuangyuan.ycj.videolibrary.whole`；VideoPlayModule-Lite=`chuangyuan.ycj.videolibrary`），相应 Manifest 的 `package` 属性移除。
  - `MainActivity` 含 `intent-filter`，补 `android:exported="true"`（AGP 8 对带 intent-filter 的组件强制要求）；`MediaRenderService` 同带 intent-filter，补 `android:exported="false"`（仅应用内显式启动）。
  - `compileSdkVersion` 30 → **34**、`buildToolsVersion` → **34.0.0**（AGP 8.3 硬性要求 compileSdk ≥ 34）；`targetSdk` 维持 **30**（避免触发分区存储/前台服务类型等运行时行为变更，留作独立任务）。
  - 协程 `kotlinx-coroutines-android` 1.5.2 → **1.7.3**（匹配 Kotlin 1.9）。
- **已排除的隐患**：全工程无 `case R.id` 形式的 `switch`（已 `grep` 确认），故 AGP 8 "非常量 R 类"规则不会阻塞 `VideoUi`/ExoPlayer 源码模块编译。
- **仍须本地验证的编译风险**：ExoPlayer 2.15.1 源码（VideoUi/VideoPlayModule-Lite）在 compileSdk 34 下可能出现个别 API 弃用/移除导致的编译告警或错误，需在 AS 内 Sync + Make 实测；老依赖 `okgo:3.0.4`、`NotchScreenTool:0.0.1`、`Cockroach:2.0.1` 等需在阿里云/中央仓库可解析。

## 二、已知遗留（需外部资源 / 较大重构，本轮未改）

### A. 64 位 ABI 缺失（影响上架与部分盒子）
- `app/build.gradle` 只编 `armeabi-v7a`；DLNA 的 `libgit-platinum.so` 仅 `armeabi-v7a` 版本。一旦加 `arm64-v8a` 会让 DLNA 崩溃；且缺 64 位无法上架 Google Play。
- **处置**：需获取/编译 64 位版 `libgit-platinum.so`（Platinum/Cling 原生库），或替换为纯 JVM 的 DLNA 实现（如 cling 的纯 Java 版或 modern-android-dlna）。

### B. ExoPlayer 源码级 fork（锁死 2.15.1）→ 已启动 Media3 迁移 ✅（部分）
- `VideoPlayModule(-Lite)`/`VideoUi`/`ffmpeg` 是 ExoPlayer 2.15.1 源码 fork，官方已停更并迁移 `androidx.media3`。
- **本轮已做（2026-07-13）**：依赖与导入已迁到 `androidx.media3` **1.10.1**（详见下方 §8 与 `Media3迁移报告.md`）。
  - Gradle 依赖：ExoPlayer 2.15.1 AAR 全部替换为 `media3-exoplayer` / `media3-exoplayer-dash/hls/smoothstreaming` / `media3-datasource-rtmp/cronet` / `media3-common` / `media3-datasource`。
  - 源码包名：68 个含 `com.google.android.exoplayer2` 引用的文件已重映射（脚本 `media3_migrate.py`），残留 0 处（ffmpeg 的 `ext.ffmpeg` 包名刻意保留以匹配原生 JNI）。
  - `VideoUi` 源码目录 `com/google/android/exoplayer2/ui` → `androidx/media3/ui`，namespace 同步改为 `androidx.media3.ui`。
- **仍待本地编译**：源码级 API 移植（Player.Listener、DataSource Factory、prepare→setMediaSource、TrackSelector、VideoUi 的 ExoPlayer 2.15.1 UI 内部 API、ffmpeg 的 BaseRenderer/Decoder 签名）。沙盒无法编译，需在 AS 内 `Sync → Make` 按报错迭代。详见 `Media3迁移报告.md` 第三、四节。

### C. 分区存储适配（Android 11+ 会失败）
- `CommonUtil.getRootFilePath()`（DLNA 媒体服务根目录）与 `UriUtils.getPath()`（本地文件选择）仍用 `Environment.getExternalStorageDirectory()`，`targetSdk=30` 在 Android 11+ 无法访问。
- `CrashHandler` 写日志已走 `UriUtils.getRootDir()`（`getExternalFilesDir`，合规）。
- **处置**：DLNA 媒体共享需接入 `MediaStore` 或 SAF；本地文件选择需改 `DocumentFile`/`ContentResolver`。改动风险高，会波及投屏核心，留待专门重构。

### D. GlobalScope 应用级后台（低优先级）
- `LiveModel.kt` / `LiveListHolder.kt` / `ThreadTool.kt` 用 `GlobalScope.launch`（永不取消、异常可能崩进程）。这些多为应用生命周期内的后台加载/轮询，按设计应长存。
- **处置**：统一收敛到一个 `SupervisorJob` + 错误处理的顶层 `appScope`，并加 `CoroutineExceptionHandler`；不在本轮改动以免破坏后台加载链路。
